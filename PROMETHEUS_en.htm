To ensure correct data reception, you can use verification codes (followed by a repeat request), or you can use recovery codes. The latter require a large number of calculations. And, directly, they are not suitable for manual use. Verification codes require a moderate number of calculations and can be used to check the correctness of data. But they have a significant disadvantage: they cannot be used to correct the received data, but only detect an error. After the error is detected - the erroneous data is transmitted again and its error-freeness is checked. And so on until the correct result is obtained. This method has a right to life, but it has a large redundancy in comparison with restoring codes and requires a sufficient amount of manual work.

We will use still restoring codes (because of their essentially better characteristics and less volume of calculations in our particular case), instead of checking codes. But, for the sake of small amount of calculations, we will use a special variant of these codes. This will allow us to quickly and easily recover the correct data.

Any recovering code has characteristics - approximation to the maximum possible recovery (recovery boundary). And this boundary is as follows: to recover one symbol you need to pass 2 or more redundant symbols. For example, the Reed-Solomon code is on the boundary of recoverability. In other words, there are no recovery codes with greater recoverability from this code (it takes only 2 redundant symbols to recover one erroneous one for this code). And one more essential characteristic is the power of the code. Power means that a more powerful code with the same recoverability can correctly recover the correct message in more cases. Thus, Reed-Solomon code 12/10 and the same code 120/100 have different recoverability. There and there - the code has 20% redundancy. But, in the first case, the code recovers any character out of 12 received characters, and in the second case - the code can recover 10 out of 120 received characters. It is obvious that in the second case the same correcting code with the same recovering ability has more power and can correctly recover data in more cases.

We will use a code with the same maximum restoring power as the Reed-Solomon code. But, in fact, there will be no recovering code. We'll use repeated data. That means that we will not introduce redundancy of correcting codes into data transmission. But we will use minimal code power. It is necessary for the simplicity of manual work with it (otherwise we would have to count much more). Our code will not have the most outstanding recovery abilities (the code power is only 1 unit or half a symbol, and it will recover not even a block and certainly not the whole message). But it is quite suitable for real practice. Deep mixing helps to improve the result a bit. Still, in most cases our "recovering code" is close to the limit of recovery with an admissible number of errors up to half of the whole message. As a small consolation for us, there is no complicated mathematical apparatus not only for using the code, but also for constructing it.  In other words, we will use a simple "problem about generals" instead of a recovering code.

Coding principle

Channel coding in our "morse code" is characterized by its extreme simplicity: it uses the 8-coded number system. In fact, for communication itself we use 6-cry. And two numbers from the 8-cry system are used for specific needs. So to master the alphabet itself, you need to master 6 codes, each with 3 bits. This is mastered in literally a few minutes. Compare this to Morse code, where you have to master about 50 characters of variable length, each of which, on average, takes 4 dots or dashes!

Let's introduce some definitions.

We will call a sign a logical value «0» and «1». The sign «0» and «1» will be transmitted equally in time. The sign «0» is a long transmission interval. It corresponds to pressing and holding the key for 3 intervals («taa»), and 1 interval is a pause. The sign «1» is a short transmission interval. It is transmitted as a short press of 1 interval, a pause of 1 interval, a short press of 1 interval and a pause of 1 interval again («ti-ti»). The signs always closely border each other without pauses. Pauses of 1 interval in signs are necessary for «alignment» of signs in time and for auditory/visual «capture» (as in Morse code) of signs and separation of one from another. Thus, the transmission of «0» («dash») requires (as in Morse code) 3 times longer holding the key than the transmission of «1» («dot»). But, taking into account the pauses - the time to transmit «0» and «1» will be the same.

The code will be called 3 data characters and a pause of 1 character. The pause allows you to distinguish the transmission of one character from another. 3 signs are 3 bits of data. This allows us to use 3 bit coding. And this is 8 binary combinations. The values 000 («0») and 111 («7») are reserved for official use. We will never use them in the data. And, we will use only 6 values for data. In fact, to transmit and receive all the information, it is enough for us to learn only 6 3-bit values by ear (note how many times easier it is than Morse code!). Principle of message recording

To record the codes on paper, we will use other codes for writing (respectively) instead of the digits «1», «2», «3», «4», «5», «6» and «space»: «c», «|», «_», «\», «/», «~» and «+». This significantly speeds up the writing process, reducing the complexity of the lettering and reducing writing errors. In fact, writing any code is reduced to a single dash. Only «space» appears as two. The codes «\» and «/» are drawn as dashes rotated 45 degrees from vertical. You can write «c» as «(». I do not do this, because I can't see well and can confuse such a bracket with a vertical line. If you're left-handed, write it as «)». - it's easier to write it down.

If it is necessary to write a character in two codes, use merged writing: for example, writing the character «8», which we represent as two codes «\/»write without taking your pen off the paper as two lines joined together: «V». For example, «\_» is also written together without taking pen off the paper when writing both codes. If, however, we have two codes shown as «/_» - actually write the second underscore at the top, resembling the letter «Г». All codes grouped by two (except for the symbol «c» - there is no way to write it without tearing it off) should be written without taking the pen off the paper. Exceptions - repeating codes like «//», «||», etc. Writing two codes «_» - we will not write them together, but we will write the second code without a break and a little lower than the first one, like this: «-_», and we will write the vertical line too (not necessarily strictly vertically, we can write it slightly slanted (like the letter «Z» - but the main thing is that the horizontal lines should not be one above the other). Such writing will eliminate ambiguity. Codes «|_» (and similar) can be written as «L» (the most convenient way to write it) or as «Г». Both are correct and unambiguously read correctly. Write it as you prefer.

To transfer punctuation marks we will use numbers. For example, we will use the symbol «!» with the symbol «1». The symbol «?» will be replaced by the symbol «7», «,» (comma) will be replaced by the symbol «9», the symbol «:» will be replaced by the symbol «8». The symbol «|» will be passed by the symbol «5». The pause between words («space» - code «+») - is not transmitted, but takes the same time interval as the transmission of any code (not a symbol!). When receiving, we will mark it as «+». If you transmit a punctuation mark (except for the «space» character) - exclude the «space» character after it. This speeds up the transmission. If you want to transmit a number, for example, «123.55» - use the following notation: «123P55» (the symbol «P» stands for «Point». If you want to transmit the number «123.55» - use the following entry: «123C55» (the symbol «C» stands for «Comma»). If you want to transmit the record «3/5» - use «3D5» (the symbol «D» stands for «Devide»). In other words, use numbers as punctuation marks in words, and letters as punctuation marks in numbers.

We will not put a dot at the end of a sentence - we will use a special format for passing on sentences.

Avoid writing it down immediately when you receive a «space», because it can be the end of a sentence. But, the end of a sentence usually has a much longer pause (the same or longer), and besides, there is a synchronization at the beginning of the next sentence.

Before transmitting a new sentence, there may be an arbitrary pause necessary for the transmitter to rest. The sentence itself is transmitted continuously (without pauses, except for the «space» character, which takes the same time interval as any code). When receiving, each sentence is recorded from a new line.

We write the received codes horizontally, leaving a free space under them for repeated reception of the message; to the right of the received codes we leave columns for calculating the corrected values: a column for the received codes and next to it a column for the calculated codes.

But, in real messages, if a punctuation mark is transmitted, it is not necessary to pass a «space» after it. The symbol «!» is used for emotion. If you need a «.» at the end of a sentence, don't pass anything.

Our «morse code» is very strongly synchronizing (similar to self-synchronizing codes in digital communication). It is easy to distinguish word or sentence boundaries. Synchroseries allow you to navigate the parts of the message. Therefore, the transmitter does not have to follow absolutely rigid time intervals when transmitting, but he can allow some deviation from them (you can gradually change the speed of transmission - if it is tiresome to keep a constant speed). It is impossible to change the speed abruptly, because there may be group interference (several codes long) and the receiver may lose synchronization: the message will be shifted, which may cause group errors. Having started transmission of a sentence, it is not allowed to interrupt (except for the time of «transmission» by «space»). In this way our «morse code» differs significantly from Morse code.

Protocol and modes of operation

Let us introduce the definitions of a protocol. Any protocol consists of a header (descriptive part) and the message itself.

The headline may contain:

- to whom the message is intended

- Who the message is from (serves as a quick initial identification of the sender; but does not authenticate the sender)

- sender's geography

- protocol version (it may not be the first one used - then it should be specified)

- cipher number (in case of batch mode)

- message language (in transliteration)

- message encoding (if not the primary encoding is used)

- message number (random number) - used to exclude repeated reception of already received message

There may be other parameters, especially in the case of project development.

 

We propose several basic modes of operation. This is done to show that the real possibilities of this method are greater than usually accepted. If, however, someone will not have enough modes, he can develop his own. And, perhaps, will offer them for general use. I would very much like to believe that.

- Interactive or dialog (this is the basic mode for dialog between two users, and it is sufficient to master only this mode for full-fledged communication) - the message is open (not encrypted); communication is performed live with an individual recipient, as in conventional radio communication.

- Packet mode (group and personal). It is a special mode designed to provide a specific exchange between users, not aiming at dialog between users, but only achieves the transfer of secret information from one user to another and can ensure the credibility of the recipient. The recipient may have no transmitter or there may be many recipients at the same time.

 

Let's introduce the definitions:

Message initiator - a person who sends a message to another and starts the communication. It is he who determines all the characteristics of communication in the transmitted header: he chooses what mode of communication will be, with whom he will communicate, in what language, etc. The initiator of the message remains unchanged throughout the entire transmission of the message.

Transmitter - the person in communication who transmits the sentence in most modes.  In packet group mode - the person who transmits the entire message to the receiver/receivers have only the receiver.

Receiver - the person in the communication who accepts the offer. It can only send a request to the transmitter to repeat the offer, request a new offer. In batch group mode, the receiver may not have a transmitter and there may be several persons receiving the same message at the same time.

In interactive mode, the transmitter and receiver can change places during communication.

In packet mode, the transmitter and receiver remain as such throughout the message.

Working channel - a channel or frequency in the radio band used for direct communication between two users or transmission of a message.

Calling channel - a special channel or frequency in the radio band used to search for a partner and harmonize the characteristics of communication, to select a working channel. It is also used for auxiliary purposes, but never for direct communication between users.

Let us impose a strict restriction on the number of users in the message exchange: let us limit ourselves to only two users: the transmitter and the receiver. This restriction is forced for several reasons: 1) the communication protocol is significantly simplified; 2) there will be no additional slowdown in the sent sentences; 3) if not 2 but, for example, 4 users were communicating, the already slow communication would slow down even more than 2 times; 4) in most cases, communication between two users is required.

For convenience, we will distinguish codes from characters in the record by always underlining the former. For example, «777777» is the transmission of seven codes 7. Or the «transmission» of «space» will look like «+» in our record.

Interactive mode

Interactive mode is the primary mode of communication and is similar to conventional radio communication. It involves transmitting a message in separate sentences with acknowledgement of correct receipt (or request for replay) after each sentence is transmitted. Interactive mode does not involve encryption of the whole message or individual sentences. Accordingly, the cipher number in the header is not transmitted. However, geography can be used (it is not available in the batch mode).

In order to fully and conveniently communicate - master the interactive mode in the working channel (without extensions: events, mutual assistance, coding tables, etc.). You will see how easy it is.

 

Synchronization

Synchronization is additional ballast information designed to ensure that the beginning of a sentence is found and that the beginning of the repetition (and an indication of the repetition itself) of a sentence is found. Synchronization does not convey any useful information and is a useless load. Synchronization is used the same for any mode. Synchronization need never be recorded. It is only received «by ear»/visually. Learn to «feel» synchronization without explicitly counting the number of correctly received codes. This is the most difficult thing when learning our «morse code» (especially with strong interference). To do this, first receive and record synchronization with little or no interference. After that, do the same with stronger interference. Soon you will be able to receive it confidently and «by ear».

Any proposal (in project terms) always starts with synchronization. Therefore, it is more advantageous to transmit long sentences than short ones. This significantly increases the speed of information exchange and reduces the load on the transmitter and receiver. The length of the sentence is determined by the transmitter himself. He proceeds from his possibilities of continuous transmission and the logical meaning of the sentence. If the transmitter can transmit long sentences, - he can combine several real sentences into one transmitted one by putting the symbol "0" instead of a dot. If another character is used in the real sentence (e.g., "!" or "?"), - then it is possible, after its transmission, to simply continue transmitting another real sentence without chipping away at synchronization.

In transmitting and receiving, synchronization never requires recording. It is always implied and always broadcast at the right time. Below, we'll show how the users' transmission sounds on the air. And we will always record the synchronization and «transmission» of «space» (recorded as «+» «transmitted» as silence one code long) on the air to understand what is happening on the air.

Synchronization consists of two parts: coarse and fine. Coarse synchronization is very noticeable by ear and allows you to quickly and easily determine what is about to be transmitted (in most cases) exact and sentence (this is because no coarse synchronization codes are used in sentences). Normally, coarse synchronization (except for group addressing - 14 codes are used) consists of 7 codes. Synchronization is considered accepted if the number of identical codes in it is not less than 4, for example, accepted: «005X070» is considered as correctly accepted coarse synchronization «0000000». Precise synchronization is always close to coarse synchronization. It is intended to precisely determine the start of the receipt of the sentence. It always has the value «654321+» («space» after numeric codes is also considered a synchronization sign). It is also considered correctly received if the number of correct codes is at least 4, for example: «X533217» is considered a correctly accepted exact synchronization. Thus, the accepted value «005X070X533217» is correctly accepted coarse and accurate synchronization in conditions of very weak signal and strong interference.

Example of conveying synchronization and sentence and conveying sentence repetition:

«7777777654321+proposal »

«0000000654321+proposal »

The receiver sends an acknowledgement of reception or a request to retransmit the last transmitted offer. «Give a new offer, the previous one was successfully received»: synchronization «7777777» must be transmitted. If the offer was erroneous and requires a repeat - "Repeat the previous offer": synchronization «0000000». The synchronization is the same in the working and calling channels.

The "capture" reliability of our synchronization will be 4/7 (about 0.57 - both coarse and fine). This is very good and allows us to reliably recognize the beginning of a sentence even in very poor channel conditions. The theoretical limit of recoverability of correction codes is 0.5. Above this limit it is impossible to make any system work.

Message header parameters

Parameters are separated by «spaces» in the parameter header and can change positions. The parameter name contains only alphabetic characters (most often one letter), but not numbers! Language_conversation (usually in transliteration if the basic encoding table is used) - one letter - own protocol abbreviations: e.g. amateur radio jargon (Jargon): L5J - (L|J Language). Callsign and message number

For on-air addressing, any user must have a call sign. Radio amateurs use their call signs (indicating that it is an amateur radio call sign), and ordinary users - must come up with a random number of symbols (letters and numbers). And if this number will be too short - it may be someone and will be duplicated. And we need sufficient uniqueness. And, if too long - will slow down communication. Therefore, we recommend using 5-8 characters for the callsign. This small dimensionality is due to the fact that we do not need absolute uniqueness of the callsign for the whole world, and because usually work is carried out in a particular calling channel. There the number of users is small, and to ensure uniqueness is very easy. And even more so in the working channel.

The message number is also a random number of characters (letters and numbers). If you send a new message, create a new number. The recommended length is also 5-8 characters. This is due to the fact that messages are quickly forgotten, and only fresh messages are of interest. Therefore, even if after some time there will be a duplicate (accidentally by another user), - it will not lead to any consequences.

 

Working channel

The working channel is the basis of communication. Only a working channel is sufficient for successful communication between two users. In a working channel, communication is only between two users. There is no possibility of subscriber selection. A channel or frequency is selected where communication between other users is minimized. The choice can be made in advance by agreement between these users. If at the moment of a new message the working channel is occupied by someone - it is necessary to wait for its release (message completion signal or channel free for more than 30 seconds) and only then to occupy it with a new call. The working channel can be defined in advance or there can be several of them (for example, even hour - one channel, odd hour - another). This increases the probability of successful communication. In addition, the working channel can be coordinated through the calling channel.

Any communication between two specific users (whether in the working channel or in the calling channel) is best organized by time. Users agree on the time at which communication sessions will be held. It is most convenient to have at least twice a day: day and night, - as the different passage of radio waves at these times. This increases the probability of communication. If the channel is busy - just the receiver is enough to be on duty for some time to receive the call. If there is no clock, - then you have to transmit more often "by eye" time (for example, at sunset). The receiver also needs to be on duty more at the radio station. The transmitter can send a "by eye" call every 15 minutes at this time.

The receiver may not necessarily "turn on" before the transmitter. In case the receiver receives only the remainder of the call, he should ignore this call until a new one is received and only then try to "recover" from interference and react to it (if the call is addressed to him).

In the working channel, a dialog is carried out and the first time a sentence is transmitted is always twice (the second time as a repetition) with the expectation of an acknowledgement (direct or indirect) or a request for a repetition.

In a working channel, the first sentence is always the headline. The headline is always one sentence.

Never use another person's calling channel as your own working channel! Make sure that the channel is not a calling channel.

In the working channel only coarse (double) synchroseries «77777777777777» is always involved - it means the beginning of a new message and then the exact synchroseries follows. Synchroseries «00000000000000» in the working channel means the end of the dialog and after it the exact synchroseries is never transmitted.

The message can be normal (dialog, packet mode) and in the form of an event (header only).

In a working channel it is impossible for more than 2 users to communicate according to protocol rules (except for header retrieval requests).

In case of operation without a calling channel (only in the working channel) - the communication is the same as by reference to the calling channel, only the caller in the working channel transmits the whole header from the calling channel instead of the reference to the calling channel. For example, the part of the call (the beginning of communication) can look like this:

In the working channel you should record immediately after switching to the working channel and if a new message is started and from the moment of receiving the exact synchronization. If the message is not addressed to you, stop recording until the new message is received. If the quality of communication in the channel is high and important parameters in the header are transmitted twice - then recording of such header is not obligatory and you can check the error-freeness of these parameters "by ear" and determine whether the message is addressed to you. If the message is for you, record it.

A sentence is always transmitted twice with a short pause (if possible, just a "space", but no more than 30 seconds) between transmissions (preventing another user from cutting in on the transmission, while the other user is prohibited from broadcasting on the same working channel at the same time). This is to allow the receiver to check whether the offer is error-free or whether an additional repetition of the offer is necessary. An exception is a special arrangement between the transmitter and the receiver (e.g. a musical instrument is used to transmit in the audio band rather than the radio band).

Rule of thumb in dialog mode: NEVER SEND THE NEXT PROPOSAL UNTIL YOU RECEIVE CONFIRMATION OF ACCEPTANCE OF THE PREVIOUS (DIRECT OR CONSEQUENTIAL)!

Direct acknowledgement of reception: the synchroseries «7777777» is sent. Indirect confirmation of reception: if the recipient has received a correct offer, he/she can, instead of sending a confirmation code, immediately start continuing the dialog and send his/her own offer - he/she will be the transmitter and the opponent will be the recipient of the offer (must send direct or indirect confirmation of receipt of the offer). This speeds up communication in interactive (dialog) mode. If it is necessary to repeat the transmission of a sentence, a synchroserial «0000000» (request for repetition) is sent.

If the transmitter cannot receive the synchro from the receiver or has received it with significant errors, it must wait for the receiver to repeat the synchro. The receiver, if it does not become the transmitter (indirect sending of a successful acknowledgement of receipt of an offer) must, after 20 seconds, repeat the sending of an acknowledgement of receipt of the offer from the transmitter or request a retransmission of the last offer.

In interactive mode, a dialog is always carried out between two users. And each offer requires confirmation from the recipient. Until the sender receives a confirmation of receipt (direct or indirect) of the offer, he will not send a new one. He will not send even a repeat of the previous one. The entire responsibility for the correctness of receiving the offer falls on the recipient. Therefore, if the recipient considers the offer (or its part) unimportant - even if there is an error, he may not demand to send repetitions, but agree to further communication. For example, «tomorrow there will be X, and we will go fishing». The recipient received the error «X» (rather than the correct transmission, for example, «A»), but this may not be important to him. What is important is the fishing trip. So he may not need to pass the sentence again and will move on to further communication. This technique speeds up the exchange in the interactive mode, but sacrifices accuracy (it is impossible to go back to request a repeat transmission in any way, since the transmitter considers the transmitted sentence to be correct).

Call channel

The calling channel is an additional and optional means of communication and is usually used only in the radio band. It is not necessary for basic mastering of our "morse code". But, it is the calling channel that makes our "morse code" a real network and allows many users to conveniently establish communications among themselves (very similar to a trunking network, but completely manual). Calling channel uses both modes: interactive and packet. Calling channel - a special channel or frequency in the radio band, on which the receiving devices of users are on standby (receiving a message). Calling channel is by no means an emergency channel of various services (maritime, aviation, river, police, etc.). The standard Morse code SOS signal should never appear on this channel. Not only that, it should not be operated by amateur radio operators in Morse code at all. A calling channel is a non-standardized (from the point of view of emergency services) channel for the exchange of telegraph (in most cases) or telephone (very rarely) modes. The calling channel does not serve for real user communication or message passing. It is where a user communicates with several others (all or a specific one) in order to negotiate real communication with an opponent or to relay events or mutual aid. There can be many calling channels. Each user group (e.g. amateur radio operators, mushroom hunters, fishermen, etc.) can locally agree on its own calling channel. On this channel (or frequency), a correspondent is called to exchange message headers to coordinate further communication. If there is no calling channel (as for example in the case of a hammer and a wall - the network becomes impossible and is considered to be the work of only two users in the working channel: all communication characteristics are specified in the header as in the header of the calling channel and no reference to the message_number from the calling channel is created or used). In the case of a laser pointer, different groups of users can shine into different parts of the sky and these parts can be considered as different calling channels for different groups as well as working channels.

The call channel is created to simplify the interaction of a large number of users. Therefore, it is built according to a simplified scheme. This implies lower noise immunity. Messages in it are always transmitted three times. If you managed to establish communication in the calling channel - you can always continue communication in the working channel, because the number of repetitions in it is unlimited (but with their increase reduces the speed of communication and increases the complexity of error correction). On the other hand, if you can not establish communication with a subscriber through the calling channel, - you can only use the mechanism of mutual assistance. Or, in general, with a particular subscriber to establish a predetermined time of communication in the working channel.

In the calling channel, headers are transmitted, but no real user communication takes place. In the calling channel it is possible to address both a specific user with coarse synchronization «7777777», and all users at once: with double coarse synchronization «77777777777777». Repetitions to a specific user and to all users have coarse synchronization «0000000». Repetitions in the calling channel are transmitted immediately after the "space" after the sentence. The number of repetitions is always 2. So, having obtained a rough synchronization of the repetition, we know that it refers to the last transmitted user sentence. Thus, the header is always transmitted with one sentence and always 3 times (except for the distress signal directly from the distressed person - it is transmitted 5 times as a mutual aid signal). In addition, there are no recipient requests to repeat a sentence («0000000»)) or to transmit a new one («7777777») in the calling channel. There is no double coarse synchronization by itself («00000000000000») in the calling channel. It is only possible in conjunction with precise synchronization. These three features immediately distinguish a calling channel from a working channel.

The baud rate in the calling channel should not differ too much from the average baud rate and is not adjustable. Therefore, try to keep to the accepted rate of transmission of messages in the channel.

The message header is always transmitted with a standard pause of one «space» (code «+») between repetitions. Encryption

For fast and convenient encryption and decryption, we will Cardan's lattice. But not a square one, but a hexagonal one. For us it is much more convenient, because we use the hexadecimal system for data transmission. Some disadvantage for practice is the rigid length of the message. For example, for a square lattice 10x10=100 characters, 11x11=121 characters. We can see the rigid message size. For a hexagonal lattice, the message length is: L=N*N*6, where N is the number of characters on one side of the lattice. Thus, with 3 characters on a side of the lattice, the length of the message will be: 3*3*3*6=54 characters. At 4 symbols - 4*4*6=96 symbols. At 5 characters - 5*5*6=150. At 6 characters - 6*6*6=216. We see always even numbers. This is a convenience for us, because it allows us to pass the password (or grid code) when we exchange symbols, which are groups of 2 codes each.

The sender and receiver of the message agree on a grid code (password) in advance.

For example, let's take the message length as a lattice with a side length of 4 characters. For it, the message length will be 96 characters (letters or digits). To specify which code out of 6 is used for the first character - take the first half of the first character of the password. For the second character - the second half of the first character of the password. And so on. Thus, to encode 96 characters we need to make 16 holes (triangular) and we need a password with the length of 16/2=8 characters (one character contains two hexadecimal codes). Users exchange the password among themselves in advance or by some other channel. The password is generated randomly. But, it is best (but not necessarily) to avoid completely random generation and to use such cells on the lattice that three neighboring cells do not contact each other (groups are no more than two). In addition, we should exclude such a variant when three triangular holes are formed around one triangle. Of course, this somewhat reduces safety, but it is still acceptable. Such techniques allow to increase the strength of the lattice and will be clearer to separate the characters during encoding and decoding. And already from the invented lattice to write off the password and pass it to the opponent. The most convenient way is to mark up on one side of the lattice and use it on the reverse (clean) side. It is necessary to draw the key of the lattice - draw a line at the top of one of the sides - from which the construction of the lattice began (the initial position of the lattice - upwards with the key). Then, just turn the grid clockwise (5 times: the grid has 6 sides).

How to get a template for coding (create a hexagon). To do this, you first need to draw a grid. The most convenient way is to fold a sheet of paper in half across. Then from the center (former center of the sheet) - the starting point, fold our sheet in three times. It is not difficult to do, selecting the lengths of each side to obtain their equality. Now it is enough to cut the resulting sector with scissors (perpendicular to the ray passing from the center to the middle of our sector). It is desirable to cut so as to get the maximum size of the hexagon. But, you can determine the most convenient size of the grid yourself. After unfolding the result of your work, you will get a regular hexagon (bee honeycomb). Now you need to color it into triangles depending on the size of the lattice. For example, we want to make a lattice with a side of 4 triangles (this will make 96 symbols in the lattice). Draw (under the ruler) 3 bold lines through the center of the hexagon from opposite corners of the lattice. It's best to make the lattice one size larger than we want it to be - this will make it much stiffer. So if we want to have a lattice side of 4 triangles, break each of the sides of the hexagon into 5 pieces. Draw regular (not bold) lines (under the ruler), parallel to the 3 previously drawn, connecting the marked parts with lines between them. You have a hexagon drawn with triangles. Cross out or sketch the outer row so that you are left with only the inner hexagon with the dimension you need (in our case - 4 elements of its side).

We see 6 sectors on the grid (marked with bold lines). Mark one sector with a bold line along the top edge of the hexagon. This will be our reference point (key). Let's turn it to the top of the hexagon. Now let's number our sector from left to right and from top to center with numbers from 1 to 16 (in our case, the length of the message is 96 characters). We will do the numbering along the horizontal lines. If we have triangles «jumping» - do not pay attention to it. For example, the first row will have (in our case) numbers 1,2,3,4,5,6,7. The second row would have 8,9,10,11,12. The third row is 13,14,15. Fourth row - 16.  Do the same with each of the 6 sectors, turning the grid (the numbering should be repeated in exactly the same order). Now we have a grid marked with digits. We need to put the obtained password (grid code) on it. For example, we have the password (grid code) OUHVLYEY or in numeric form: 2561351462163416.  We take the first code of the first character of the password (the character is represented by two codes during transmission). In our example it is the code «2». Count clockwise from the key, placing the grid with the key upwards (starting from it) the second digit «1». Draw this cell (triangle). Take the second password code (code «5») and do the same for it - look for the fifth digit «2» (from the key). We do the same with the whole password. We have 16 sketched triangles. Now we cut them out. After that we turn our template over with the face (clean) side to ourselves and transfer the key to the front side (mark also the same edge of the grid as on the back side). It is best to outline the triangular holes on the front side with a contrasting pen or felt-tip pen - this will speed up the identification of the grille holes and eliminate mistakes. We got a template for encoding and decoding our messages 96 characters long. Even better, make a template out of cardboard or stiff paper. On the back side, write the group number (if necessary, if you are in a group) or the opponent number (if necessary, if you are not in a group) and the cipher number (if necessary, for example, this cipher is number 4 for packet mode).

On paper, it is better to put the dashes (in two opposite corners) of the grid and align the grid with them when rotating. This is necessary to avoid displacement of the grid and allows you to eliminate the writing of symbols not in their places.

To transmit an encrypted message, it must first be encrypted using the grid. To do this, place the grid on a blank piece of paper with the key up. Make marks near the lattice on the sheet of paper (we will use them to align the lattice when it is rotated). Write text one character in each triangular hole in the lattice from top to bottom and from left to right, on horizontal lines, even if the triangles are «jumping» (they are placed in a line at intervals with the base down and up). If a «space» is needed, enter the «+» sign. If there is an end to a sentence - the symbol «0». Punctuation marks are standard. If the grid has run out - turn it clockwise by one sector (aligning risks). Keep doing this until the message is completely filled. If your encrypted message is smaller than the grid size, then fill the rest with random characters - it is guaranteed not to weaken the reliability of encryption. Although you can fill simply with «+» signs (here reliability may be lower, but higher - convenience and speed of transmission; but, if, you have a lot of empty space remains - there is a high probability when transmitting to have several signs "+" in a row - in this case, a loss of synchronization is possible and you can get a shift of signs; so the method is suitable if there are few unfilled characters in the lattice; as an alternative - you can fill the empty cells with a character, such as «0» - in this case, loss of synchronization will never happen). Now you can transmit an encrypted message (in a row from top to bottom and from left to right) without the grid.

Before receiving the message, we need to make a template on the grid (mark the triangular cells). To do this, we put the grid on a blank piece of paper. We trace the perimeter of the lattice. On the edges we make dashes near each triangular cell. Now we need to draw the triangles with a ruler and pen. The template for receiving the message for the lattice of the required dimension is ready. At reception we write the message character by character (one symbol in each triangle) directly along the lines of the lattice: from top to bottom and from left to right. To decode, we just put the grid with the key up on the written pattern. Now we simply write out the message top-down and left-right along the open symbols, turning the grid clockwise 5 times.

 

Events

Events are a very convenient mechanism that allows you to send a short message only to a specific user (always in the message header). And possibly receive a response only from him/her. Events are not related to any particular mode. They are used to inform the user that some predetermined action has occurred. Examples of events are: "Proceed to phase 2 of the operation"; "Return to point 1". Events, unlike messages, cannot be transmitted in text. They are transmitted by a short number. Advantages: very compact transmission; identified sender; intermediary can be used; sender can be notified of delivery to an identified recipient; no encryption required. Disadvantage: only certain codes can be transmitted in advance (at the stage of agreement between sender and receiver).

Events are a separate mechanism (an add-on to both the worker channel and the caller channel) designed to conveniently solve a simple problem: inform the sender and receiver (each other) of their default states, e.g., I am in state «2». The events are of interest for three reasons: first these messages are very short. Second, the sender and receiver are identifiable. Third, events can invoke a mutual aid mechanism for the case where the sender and receiver cannot contact each other. In this case, it is also impossible to talk about the guaranteed passage of the event, because the mutual aid mechanism does not guarantee anything.We need to keep in mind that an event in the calling channel (unlike an event in the working channel) is a non-guaranteed delivery mechanism. It can always fail to be received by the recipient. And, even through mutual aid. So, coordinate with the recipient the time you send the event - to increase the probability of receiving it. But, if the event is received, the recipient knows for sure that the event was from the right sender and no one else can tamper with it or change it.

Events can be single-phase, two-phase and three-phase. It is up to the sender to determine which one he will use.

A single-phase event is a notification. It does not imply any response from the recipient.

A two-phase event is the same notification (as a single-phase event), only with a delivery notification. If the user has received a delivery notification - he knows that the right recipient is guaranteed to receive his event.

A three-phase event is an extended two-phase event in which the event sender received notification of the recipient's status and its response to the sender's command. In return, the event receiver received notification of the delivery of its response to the sender.

The number of event phases is displayed as the parameter name: single-phase – «M» (one character «M»); two-phase – «MM» (two characters «M»); three-phase – «MMM» (three characters «M»).

The current phase of the event is displayed with a single number character after the number of phases in the event. For example, «MMM2» indicates that the current phase is 2 in a three-phase event.

The same event has the same message number. Also, if an event has more than one phase, the event has the same event number and the event phase numbers differ. The number of phases and the current phase and event number are placed in the event itself, which is always placed in the message header.

The event has a continuously increasing number. After a successful event reception (authentication), the receiver writes this number in his notebook, crossing out the old one. If the new event has a number less than or equal to the old one - the event is not even worth not only processing (checking), but even accepting.

A «+» («space») sign is never passed in the event, as it is a header parameter.

The sender and the receiver generate a random message (a random set of symbols, for example 96 symbols) and keys (for example, for a grid of 96 sims the key length should be 96/6=16 symbols, and the random number - 16/2=8 symbols) in advance (for each status and command value - it is best to come up with the maximum number of numbers (and keys paired with them) and each status and command is assigned a number for each phase). Each event has its own unique number. If the number is greater than the previous successfully processed event - it is accepted for processing. If the processing was successful - the old number in the notebook of the accepted event is overwritten by the new one. The transmitter adds up all the numbers of the number. Now he comes up with a random number (grid dimension divided by 6 and divided by 2). For example, for status we need to take such a grid (with the desired password, the number we want to send) and count the folded number by holes. From this hole we will count (skip) the number of holes of the first half of the random number. We take this character - write it as on the transmission. Take the second half of the first number and from the taken symbol count this number of skips - take the necessary symbol as for transmission. And so with all numbers of the random number. We also do the same for commands. Thus, we have formed an event for transmission. On reception - all the same, but first decode the event - the status and the command (search for all the lattices: if an error - go to the next), and in case of success - form the response in the same way. And so on for each phase of the event.

Consider an algorithm for encoding and decoding an event.

Attention! All algorithms related to encryption are recommended. But, usually, they are good enough for practical use and do not need to be changed. However, you have the right to change them at your discretion. The main thing is that the receiver should perform the same actions as the transmitter. No one else will be affected by changing the algorithm.

To transfer events between specific users - one single grid is created and filled with random values (random «message»). All this is put into the notebooks of both users. Now they can send events to each other. The initiator can be either of them.

Distress signal

The «SOS» signal (for voice communication, the analog of the «SOS» signal is «mayday») has no Morse code entry. It is a separate character without letter intervals («...---...») - and it cannot occur in our "morse code". This will eliminate false reactions of Morse code connoisseurs to involuntary parcels. Even «S O S» («... --- ...») cannot be transmitted in our "morse code".

It should be noted that in the practice of radio exchange, the «SOS» signal was not transmitted by itself, but was part of the mandatory group of signals transmitted by the radio operator or automatic emergency transmitter, with which any ship was equipped. This group consisted of the following signs: general call signal «CQ», transmitted 3 times, «DE» - (this is I) representation, then went the ship's call sign, then the «SOS» signal, transmitted 3 times, then the latitude and longitude of the ship, and finally 2 dashes lasting 12 seconds each for direction finding.

If you are in distress, please use the standard emergency frequencies and Morse code to signal your distress. We will support additionally the same analog of the «SOS» signal in our network, but this signal does not change the standard one in any way. It is an additional means (but not a substitute for the standard «SOS» signal!) and may be the last hope for a person in distress if the standard signal cannot reach its destination. Our «distress signal» is longer-range, but the number of recipients may be much smaller! We will use our helpdesk to send it. Our signal (let's call it «_sos_») will be transmitted as a standard header parameter. Only the number of repetitions instead of the standard 2 (1 first take and 2 repetitions) should be done 4 times (1 first transmission and 4 repetitions). The transmission of the «_sos_» signal from a person in distress may be made more than once. This is at the discretion of the person making the distress call. This will increase the range and probability of receiving the signal. If the «_sos_» signal is transmitted in the calling channel, all other users should facilitate its transmission as much as possible: temporarily free the calling channel. If the user is able, he/she should respond to the request for help and notify the rescue services. If the user is unable to contact the emergency services, he/she should/can join the mutual aid mode if the number of repetitions is 4 or more (the message comes directly from the person in distress, not from other mutual aid participants) and repeat this message (2 repetitions, the message is literally word for word) on the air from his/her transmitter. If you see that the number of «_sos_» messages from mutual aid participants with the same text is large enough (more than 10) - you may not transmit it yourself. This is already enough for someone to call the rescuers. Note that the person sending the «_sos_» signal is not identified, just as in the standard «SOS» signal. This is not possible. Otherwise, in a real danger, not everyone would be able to inform about it. Unlike our «_sos_» signal, sending a standard «SOS» signal out of order is a criminal offense in most countries of the world. Sending our «_sos_» signal without a real case of distress signaling is not a criminal offense. This means that people can start to abuse this action. In addition to this - in our case, mutual assistance creates an "information storm" - it simply throws the calling channel out of balance. If the «_sos_» signal starts to be abused, I recommend to give it up altogether. After all, "matches are not toys for children". Anyone can set fire to the forest!

Packet mode

Packet mode is a specific mode. There is no dialog at all. There is only sending a message by the sender and receiving it by the receiver. And the message must be encrypted. That's all. Batch mode contains two real modes: group and personal.

- Packet mode - the message is closed (encrypted) and the recipient(s) must have a decryption key; the message is pre-prepared and transmitted already unchanged:

- Group - the message does not imply a response from the receiver (the receiver can only have a receiver without a transmitter); there can be any number of receivers at the same time; the message can be pre-recorded and played back automatically (if technical means allow).

- Personal - the message is sent to an individual recipient; the recipient acknowledges receipt of the message (with recipient authentication).

Previously, we have considered the interactive mode. In the packet mode all the above described are applicable: synchronization, work channel, calling channel. The calling channel is used together with the interactive mode. This means that the mechanism of events and mutual assistance is also possible - it does not depend on the mode of operation. Actually, the main indicator of packet mode is the presence of encryption indication in the message header (it does not matter whether it is in the calling channel or in the working channel).

Packet mode - involves transmission of the entire message in two sub-modes: without acknowledgement (the receiving party has only a receiver, but no transmitter), group reception is possible; with acknowledgement of reception/error. When receiving a message with confirmation of reception - in an encrypted message, a sentence is exactly 1/6th of the whole message.

In the encrypted message of the packet mode, we will refuse to divide the message transmission into sentences and introduce synchronization symbols on them, as in the interactive mode. Instead, we will divide the whole message into 6 parts. And each part will be transmitted as a sentence in the interactive mode (with synchronization). This is very convenient for practical use.

So, if we have removed synchronization on real sentences, we need to introduce a «dot» symbol on real sentences. We'll use the «0» symbol. «Space» will be transmitted as a pause as well as in interactive mode, and will be recorded as a «+» symbol.

Addressing to a user can be direct (by his number) or indirect - through the group he is a member of. The number of groups a user can enter is unlimited. A user (as well as a group) can have more than one password (respectively, an encryption/decryption grid). Each password for a pair of users or a group has a number.

The number of ciphers a user has for each group or a particular opponent can be several. And the grid sizes can also be different. The user chooses the most convenient grid size for transmission.

If the message size is smaller than the grid size - it is desirable to fill the remainder with random character values. Alternatively, you can fill it with «0» characters. This slightly reduces the resistance against tampering.

Of course, it is impossible to achieve such a level of encryption, identification, integrity verification, protection against spoofing as on a computer manually. And, in fact, it is not necessary. But, nevertheless, practical application of the method is possible.

 

Packet group mode

Group mode is a rather simple and interesting mode in practice. The advantages of this mode: the message is always encrypted and only those who have the encryption key can decrypt it. Moreover, the receivers only need to have a receiver and do not need a transmitter. Its disadvantage is that there is no guarantee that the addressee will receive the message, and even in case of successful receipt, the sender does not know about it either.

The group mode allows you to send a message not only to a single recipient (field «T|» contains the address of the recipient), but also to a whole group (field «GT|» contains the address of the group of recipients).

The packet mode header contains the following: recipient_number (T|) or group_number (GT|) - to whom, sender_number (F|), cipher_number (S|). The header must be passed 5 times, and only after that the message itself must be passed. Essential fields can be written twice (to whom, from whom, message number, encryption key). You may not repeat sentences in the message at all. This will greatly simplify the recording of the message (e.g., on a tape recorder), reduce its duration and recording errors. When receiving a message, it is enough to make sure that the recipient's address, the sender's address and the desired message number are correct (in principle, only the last one is enough). If everything matches - receive the message again and use "restore" with the previous one. And so on until the errors are completely eliminated or the sender has exhausted repeating the message (in this case there may be no error-free reception).

Packet personal mode

This mode is more complex than group mode. It also supports mandatory encryption. But, the message can only be sent to one recipient. The recipient requires a mandatory receiver and transmitter. In return, the mode offers the assurance that the encrypted message will be received reliably and without error by the recipient. And it is by an identified receiver. That is, the sender, if he has received confirmation of identification, can always tell that the right receiver has received his message.

The reception method resembles a somewhat interactive mode, but in return the receiver cannot send anything to the sender except an acknowledgement of receipt of the offer (or a replay request). In response, the receiver sends an identification confirmation of receipt of the correct message (one identification confirmation for the whole message). This, only from the look of it, seems to be an interactive communication. In fact - the communication looks more like a batch group mode (with seemingly dialog elements). The identification confirmation is sent as a new dialog (message) from the recipient of the encrypted message to the sender with the same message number.

Here we need to refuse to process a packet that we have already processed before. This is necessary to prevent an attacker from re-accepting the packet (an attacker cannot send his own message, he can only re-send a previously intercepted one). The verification method is a combination of the event and group packet mode methods. But, once again, we emphasize that the handling of any encrypted methods is advisory: you are free to change the method at your own discretion. This does not affect others in any way. The main thing is that the receiver should be able to process the message correctly (use the same method as the sender).

Encoding and decoding automation methods

Methods of acceleration of coding and decoding of our "morse code" signals are reduced to replacement of manual telegraph key with mechanical or electronic one. At the same time it is desirable to change the very principle of frequency manipulation (convenient for a human) to other methods, which are more convenient for technology and allow to achieve higher speeds of work.

If we use the same transmitter and receiver as in conventional Morse code operation, we cannot change the modulation of the devices. Instead, we use only manipulation. We will consider this particular approach.

Mechanical manipulators

It is possible to further develop our "morse code" - if we replace our manipulation instead of frequency (we transmitted bit "0" as a long dash with a length of 3 transmission cycles and a pause of 1 cycle, and bit "1" as two dots, each with 1 cycle of transmission and 1 cycle of pause) with a phase manipulation. Moreover, we do not change the physical modulation of the transmitter: as it was CW, so it remains - only the bit interpretation changes. Phase manipulation, as well as modulation, is poorly determined by human hearing, but much better by technical means than frequency manipulation (the requirements for manufacturing accuracy will be at least 2 times less, and the speed of transmission and reception will be 2 times greater). For phase manipulation bits will not be divided into 4 intervals (as in frequency), but into 2. Bit "0" will be transmitted as "-=", and bit "1" - as "=-". We will keep the guard interval in each code (previously we "passed" the blank after 3 informational bits 1 bit of blank as "----"), but now it will be as of two intervals, just like the informational ones: "--". We now have the freedom to encode and decode our codes. Let's consider, how it is possible to achieve very original mechanical encoding and decoding of codes. And very fast, and obviously inaccessible for the speed of encoding/decoding by ear.

Let us first consider the decoding principle. To do this, let's replace the physical speaker on the receiver with a white LED. But not one, but two. Take a transparent ruler and cut out a piece of it with a width of 5-10 mm and a length of 5-10 cm. Glue transparent glue LEDs on the two shortest ends of our "light guide". Let's paint our LEDs and glue with black felt-tip pen and all our "light guide", except for one end of the "light guide" (the end 1-2 mm thick and 4-8 cm wide) will go to our indicator. This end of the "light guide" should also be painted 1 cm from the sides of the LEDs. Now we need to divide the entire unpainted end of the "light guide" into several separate "bulbs". To do this, we leave our "bulbs" with a width of 3 mm. We divide all our "bulbs" into groups: on the left we make 2 "bulbs" (between them 2 mm painted with a black felt-tip pen), and the "bulbs" painted with a transparent felt-tip pen of magenta color - will indicate synchronization (one shows - down, the other - up). The next group also consists of 2 "bulbs" - paint them with transparent yellow color - they will indicate the reception of codes 0 and 7. And the right group of 6 "bulbs" are information bulbs. We paint them with transparent blue felt-tip pen. Now you need to solder LEDs counter: plus to the minus of each of them (this will achieve a triple effect: to get protection on the reverse voltage for each of the LEDs - they protect each other; to get doubling the frequency of flashes, for example, from 500 Hz to 1000 Hz, which will provide a smoother glow and reduces the stroboscopic effect; doubling the brightness of the glow with a more uniform illumination). Place our "light guide" inside a white matte (or transparent) plastic cylinder (hydrogen peroxide vial or similar). Fix the bubble on the axle. This will be the receiving part of the drum. Connect our LEDs instead of the receiver speaker through a current limiting resistor. Try to turn on the reception and catch some radio station. Our multicolored "bulbs" should flash simultaneously. Now we need to mark our receiving drum into "disks". To do this, we rotate it a little at a time and lightly touch it with a pencil in the places where our "bulbs" glow. We got 12 "disks". Now we need to mark each "disk" according to its purpose. The two purple disks are synchronization in the rotation of our drum. In our manipulation, the middle of each bit is the fulcrum (there is a drop). And during one revolution of the drum 3 information bits are transferred and one bit is used as a "protective" bit (for synchronization by codes). So, we have 3 synchronization flashes per revolution of the drum. For synchronization, we leave a 2 mm long window unpainted on each quarter of the drum from the middle of the bit (where we always have a drop in the bit). All the rest is painted with a black felt-tip pen. We do this with both synchronization disks. The only difference between them will be that on the first disk we "shift" the "windows" to one side from the center of the bit, and in the second - to the second side (the shift will be about 2 mm one "window" of the "disk" relative to the other "disk". Now, when rotating the drum, we only need to strive for the same brightness of both synchronization "bulbs", without paying attention to their own brightness. If one light is dimmer than the other - you need to speed up or slow down, depending on which light is dimmer (choose yourself what you prefer to count - brighter or dimmer, you can do both). This will correspond to the approximate location of the synchronization at the middle of each bit. Information and 0 and 7 codes are also colored according to their values. Now, when rotating the drum and receiving "morzianka" we will flash "bulbs" twice for one information bit and in the place where the marking bits coincide with the transmitted ones the greatest brightness will be observed. We have 8 "disks" with brightness (of course, averaged, due to the luminescence rate: at high speed of disk rotation the LED will shine longer for a more "bright" code and will be averaged by the human eye) will be in 3/3 - 1 disk, in 2/3 - 3 disks, 1/3 - 3 disks and extinguished - one disk. Now, even with a very fast rotation of the knob, it is enough to identify the brightest disk. This will be the correctly accepted code. You can build the approach inversely: mark the disks on the contrary and take the completely "extinguished" disk as the correct code. But it seems to me more difficult to perceive it against the background of glowing disks. In order to reduce the influence of phasing error on the reception of correct information and to reduce false alarms, you can use the well-known in printing technique "trapping" ("trapping" - you can look it up in wikipedia). In this case you need to darken disks not exactly in the right places, but a little bit (for example, 20 percent) - trapping on the light parts. Now, even with a drum rotation error up to 20 percent, we will not "light up" the wrongly received code with lies. Of course, all signals (including the useful one) will slightly decrease in brightness. Even a layman can take in information very quickly with this method.

The transmitter is built on the same drum as the receiver. But it is placed on the left side of the drum (the rotary knob is also on the left side to make it easier for the right hand to write down the received codes or "transmit" them). The transmitter is built on the principle of "sharmanka": it consists of 9 mechanical disks, which "press" the transmitter key at the selected code. To do this, we take 9 thick washers (say, 20 mm in diameter and 2.5 mm thick) and 10 thin washers (about 0.5 mm) of the same diameter. Now, on each of the 8 thick washers, we sharpen 3 of the 4 sides of the washer on the sharpening machine in accordance with the desired code. Leave a quarter of each of the thick washers untouched. Now put our washers on the drum axle, alternating with the thin washers. Rotate the thick washers so that their untouched quarters match. And we have a quarter of all disks forming an even surface. Now it is enough to pass a thick 2 mm copper wire under the disks, fixing it to the axle on one side and making a handle on the other. We thin disks will allow us to move the handle freely only in the required quarter turn of the drum, but will not allow the wire to stop on them. It is highly desirable to place this quarter in such a way that the moment of moving the code handle comes at the downward rotation of the drum handle. This increases stability on the table of the device. When turning the drum, the desired disk presses the wire down ("bends" it). It is best to make a loop on the wire (by twisting the wire around a pencil), and make it flat at the bottom (bend it with pliers). Then, if you put a contact in the ring of the loop, then when you push the drum on the wire, the contact will open. This is what we need, because when we pass the "pause" we have a raw disk and it will open the contact. Now, by turning the drum knob evenly enough, it is enough to move the code knob in accordance with the transmitted codes during the desired quarter of the drum rotation (reminiscent of how chords change in guitar playing). Copper wire is much better than steel wire: in combination with steel washers, it has the property of antistearability. In other words, it has almost no friction wear and requires no lubrication at all. This is similar to the operation of brushes in collector electric motors or the operation of slip rings in trolley buses. Only here, the electrical conductivity of copper is not really needed. Some lack, in this case, is the mechanical softness of the material. Alternatively, copper-plated steel wire can be used.

Other methods of mechanical reception are also possible: for example, without a protective pause in the transmission of codes. To be able to work with such codes it is necessary to use a "high-precision generator" - for example, a pendulum J. Advantage - 25 percent higher transmission and reception speed than when using pause in codes. The principle of reception is the same as with the previously described drum, but now we use a disk. We calibrate the pendulum to a hard receive rate (and transmit rate as well) by varying the length of the suspension string. Calibration should be done by a regular clock with a second hand. We make the pendulum swinging exactly in a circle. For this purpose it is enough to make a slit in the disk from the inner side, but half the length of the disk (and grind this slit to smoothness). Set the disk on the needle for easy rotation. Tape the very end of the inside of the slot to prevent the pendulum string from falling out. Mark the marks on the slot as min and poppy (you can just paint the disk in this range in green or any other color, or just draw circles). The principle of marking and reception is the same as in the drum method: we use the half of the disk without a slot as virtual "disks" and place a ruler with an LED under the disk itself in a place not under the slot. The second LED (as in the drum method can be used, or you can do without it, because the stability of oscillations here is high and synchronization is carried out separately). Now we pass the thread of the pendulum into the slot (the weight of the pendulum under the disk, but the weight should lie on the disk from below, when the thread almost touches the end of the slot in the center of the disk - this will eliminate the swing of the weight under the disk when the disk is spun), fix it and calibrate it to the desired frequency: just spin the disk and the pendulum will almost bump the thread at the end of the slot in the middle of the disk (the weight will lie on the disk from below), wait until the thread moves away from the end of the slot and becomes in the working range. Now count the number of oscillations of the pendulum (they will be exactly circular) with the disk. And by changing the length of the thread we achieve the maximum coincidence with the clock. Due to the initial synchronization we determine the beginning of our packet (here we receive the whole packet, not one or two codes) - we rotate our LEDs under the pendulum until we get the desired synchroseries (when viewed from above on the disk). Such synchronization should be done every time when receiving a packet. The heavier the load - the slower the pendulum will come to the working range, but it will stay in the working range longer. It should be taken into account that the accuracy of mechanical devices in any case is relatively small (an ordinary mechanical clock for a day can "go away" by 1 minute, and this is in the neighborhood of 1500 oscillations).

Other mechanical methods can also be proposed. For example, drip (on the principle of water clock) - one drop - one code. The drop closes (conducts electric current) the contact and the bit is transmitted. A new drop is dropped only when the previous drop has completely worked off the code transmission. The disadvantage of the method is that the device cannot operate at sub-zero temperatures. The air method is also possible: the shock wave propagates in the air with a fixed speed (slightly less than the sound speed). It is enough to hit the end of the pipe with a hand and a sound wave will appear in it. Now let us place holes in the pipe and pressure will be induced in them. If the pipe is made in the form of a snail, the holes can be placed next to each other. Now when you press the required code you can close the required holes and the required code will be generated. The reception can be realized as in the drum principle by the same holes. It can be struck with a speaker instead of with the hand. Even greater possibilities can be achieved by the chamberton method, which is the most perfect resonator (approaching in quality to the quartz resonator in electronics) in mechanical engineering. The most promising combination in combination with moiré: it allows to exclude lens systems and use directly the LED. Electromagnetic systems can be used instead of optical systems: magnet and coil. When using resonators for highly stable oscillations, it is necessary to abandon the guard interval for code transmission.

Electronic manipulators and modulators

Electronic methods give a much greater flight of fancy for the construction of transmitters and receivers. But, if we have the possibility to change the modulation of devices, we can achieve much better results (combining with manipulation) in terms of reliability, speed, range. It is possible to do including direct modulation (for example, frequency or phase or combined with amplitude) of the carrier. This allows the use of phase frequency adjustment (if the carrier is broadcast continuously, even in pauses) and allows you to achieve better performance in terms of range, stability of communication and simplicity of devices.

The most promising is the transition from primitive CW modulation to the use of linear frequency modulation (LFM) in an all-electronic coding-decoding method. Something similar to LoRa network modulation (in LoRa - for code transmission - one "chirp" is used and the phase change of this "chirp" is determined, while here - many). This modulation allows to achieve several goals: it is very noise immune and long range; it requires less bandwidth than conventional frequency or phase modulation; it allows to use phase auto-tuning of frequency in receivers and to abandon quartz stabilization and thermal stabilization altogether. For this purpose it is enough to use 9 frequencies of change of "chirp": 8 information codes and 1 code of presence of "silence" (presence of transmitter broadcasting - it is better to put some separately from information LEDs). "Silence" should be present after pressing each information symbol and serves to synchronize the transmission of codes (the duration of "silence" can be offered in half the duration of the information code). The format can be organized as follows: "silence" is broadcast at a rate of 10 "chirps" per second (here, the frequency of "chirps" should be quite significantly higher than the speed of pressing the information buttons, but at the same time it is desirable not to make its frequency too different because of the significant expansion of the spectrum, and hence the reduction of "range"). And information buttons - from 20 to 90. Thus, on the receiver it is enough to select the desired frequency of "chirp" - this will be our code. On the transmitter - just need to generate a "chirp" (change the rate of change of frequency of the transmitter - similar to the change of speed of the "siren" in the ambulance) according to the pressed code (from 0 to 7) or "chirp" with 10 hertz with unpressed buttons, but turned on transmitter. But some difficulty arises compared to use in LoRa or in FT8 (the latter does not use LFM, but is also based on fixed symbol duration) - variable bit symbol length due to human timing counts. This is especially important in poor reception conditions: it allows the useful signal to accumulate over time and to be emphasized in conditions of superior noise. Since in our case it is a human, not electronics, who transmits and receives, the duration of the bit on reception will be different. It is possible to almost compensate for the different durations by a combination of electronics and humans: electronics, for example, detects the duration of a tone that exceeds 0.2 seconds. If, for example, the transmitter presses buttons once per second, the receiver may not correctly capture (under interference conditions) the code. After the interference passes, the code may be different (most likely correct). And it will be captured for a longer time than the interference or incorrect code. So, you need to ignore blinking diodes too often when receiving: you need about 1 second or more. In this way we combine the advantages of electronics and human eye in order to significantly simplify the electronic path. This scheme allows to build a fairly simple transmitter and receiver (only on simple analog equipment without digital circuits), allowing to work in a significantly noisy communication channel, while having the comfort of almost digital transmission and reception (not fully automatic). Some disadvantage is complete incompatibility with standard communications (but, FT8 is also only "self" compatible, as is LoRa).

 

Here we assumed operation of all systems in half-duplex (or simplex for packet group mode) mode of combined or spaced reception (for repeater operation). Operation in duplex does not make physical sense, since the transmitted "digital" information is not perceived by the human ear as voice information and requires efforts to process the information from a human.

We originally developed our "morse code" to work on absolutely any devices without requiring any modification. Besides, we completely excluded microprocessor devices for information processing. If such possibilities are used - the development will resemble LoRa or FT8 or something similar. Because there is a clear tendency to replace manual labor with automation. The result will be in many ways better than today's development. But that's a separate topic and a separate project. If anyone needs it. For there are many alternatives in the world. Each of them has its advantages and disadvantages. But still, our work today is special.
